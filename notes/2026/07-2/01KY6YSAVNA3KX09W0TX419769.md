Follow-up from DEV-574 / DEV-576. Status and Priority are currently edited through the generic "add taxonomy" (+) picker in a note's taxonomy section — so they're not visible by default and a value can only be set after adding the taxonomy row. For something *every* task/project has, that's clunky and easy to miss (it read as "I set priority but nothing shows").

Make **Status** and **Priority** first-class, **permanently visible** fields, like Type/Project:

**Checklist**

- [ ] In the note editor, show **Task Status + Priority** (Tasks) / **Project Status** (Projects) as always-visible controls in the **meta top bar** (alongside Type / Project), not buried in the taxonomy "+" section — settable in one click even when unset.
- [ ] In the **capture window** (create flow), include the applicable status + priority selectors when creating a Task/Project, so they can be set at creation.
- [ ] Drive them from the registry's built-in ids (`task_status` / `project_status` / `priority`) so user edits to those taxonomies (DEV-574/576) flow through; reuse the existing value pickers/colours.
- [ ] Keep them out of the generic taxonomy "+" list (so they're not offered twice).
- [ ] (Optional) set a card's status/priority directly from the Kanban (e.g. a small menu), complementing drag-to-restatus.

Pure UX/surfacing on top of the existing model — no data or ADR change. Builds on the capture taxonomy selectors (DEV-511) and the note-editor meta header.
