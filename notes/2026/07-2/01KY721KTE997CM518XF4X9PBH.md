Follow-up to DEV-887 (ADR 0034). The trash currently keeps notes until manually restored or deleted; the `trashed:` frontmatter field already records an RFC 3339 timestamp of when each note was trashed, so auto-purge needs no storage-model change.

Add a retention setting: permanently delete notes that have been in the trash longer than N days.

Design points to settle:

* **Setting + default.** Settings → Storage seems the natural home (next to the trash's other machinery). Off by default, or a generous default like 30 days? Linear uses a fixed retention; for a personal vault, opt-in may fit better.
* **When to sweep.** On app start plus a daily timer is probably enough — the purge is not time-critical. The sweep goes through `delete_note` per note (same as Empty trash, DEV-887) so a trashed project's live tasks are orphaned consistently.
* **Surfacing.** The Trash section could note the policy (e.g. "items are deleted after 30 days") and each row's remaining time could ride the existing trashed-date tooltip.
* **Multi-machine.** With git-sync, whichever machine sweeps first wins; the deletion syncs like any other. No coordination needed, but the sweep should tolerate a note already purged by another machine (delete of a missing file must be a no-op, as Empty trash already handles).
