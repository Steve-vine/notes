    Steve Vine · 2026-07-06:

    **Answer: yes, this is expected behaviour** — and the folders do fill up over time, very slowly, by design.

    **What you're seeing.** Notes are stored at `notes/<aa>/<bb>/<id>.md`, where `aa` and `bb` are the first two bytes of a **SHA-256 hash of the note's id** (ADR 0004, `shard_components()` in `src-tauri/src/note.rs`). That creates 65,536 possible leaf buckets (`256 × 256`), and the hash scatters notes across them uniformly. Your vault currently has **25 notes spread over 25 leaf folders** (23 top-level folders — two happen to share a first byte), so at this size virtually every note sits alone in its bucket. That's the statistically expected outcome, not a bug.

    **How it fills up.** Average files per leaf folder ≈ `notes ÷ 65,536`:
    - 1,000 notes → almost all folders still hold 1 file
    - 65,000 notes → ~1 file per folder on average
    - 1,000,000 notes → ~15 per folder

    So "one file per folder" will look normal for years — and that's the point: **no directory ever gets large**, which keeps OS operations, git status/sync, and the full index rebuild (walking the whole tree, ADR 0003) fast at any vault size.

    **Why hash, not id prefix?** ULIDs are time-ordered, so notes created together share a leading prefix — prefix-sharding would pile a burst of captures into one hot folder. Hashing scatters them evenly regardless of creation time (the full reasoning is in `decisions/0004-hex-sharded-storage.md` and `brief/storage-architecture.md`).

    Worth remembering the layout is purely machine-facing: you never navigate it by folder — retrieval is search and the browse tree, and finding a note's file from its id is a pure function (no lookup). No change recommended.
