Introduce the concept of **users/people** so a Task can be **assigned** to someone, for organisation (notional — no auth/accounts).

Per the design chat, this is likely "a special kind of taxonomy": an **Assignee** taxonomy whose value pool is the set of users, scoped to Task (and maybe Project for an owner). A **Users** Settings page manages that pool (add / remove / rename people).

**Open modelling question (decide first):**

* **Assignee taxonomy** (value pool = users) — fits the existing model, indexing, browse, and board filtering for free; or
* a **dedicated** `user` **entity / note Type** — heavier, but supports richer per-user data later.
  Recommend the taxonomy approach for now; note the decision in the issue (ADR if it ends up a new entity).

**Checklist**

- [ ] A **Users** Settings section: add / remove / rename users (the assignee value pool).
- [ ] An **assignee** field on Tasks (single or multi — decide), surfaced in capture + the note editor.
- [ ] Show the assignee on Kanban cards / the project task list.
- [ ] (Stretch) group/filter the Kanban by assignee.

Relates to the editable-taxonomy work (DEV-573) if assignees are managed like statuses.
