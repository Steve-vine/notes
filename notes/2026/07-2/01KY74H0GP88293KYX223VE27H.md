While the app is running, taxonomy values written to notes by an external process (MCP server) are silently dropped if the value was added to `taxonomies.yaml` after the app loaded its definitions.

Observed 2026-07-23 during the Linear migration: `imported_from: linear` was added via MCP at 07:09Z and stamped on 367 migrated tasks; at 09:08:56Z the running app re-serialised 369 note files from stale in-memory taxonomy definitions (vault commit 5a4a874c) and removed the `imported_from` line from every one. Bodies, comments, statuses, and sprint fields were untouched.

Likely fix directions:

- Reload taxonomy definitions when `taxonomies.yaml` changes on disk (file watcher), and/or
- Never drop unknown taxonomy keys/values on re-serialisation — preserve what's in frontmatter even if it fails validation against in-memory definitions (align with ADR 0045's semantic-merge stance that the serialiser must not lose data).

Repro: with the app open, `add_taxonomy_value` + `update_note` via MCP using the new value, then trigger any app-side re-save sweep of that note.

Found via dogfooding on migration day (NOT-368).
