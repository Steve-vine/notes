Add the write half of the CRUD surface to `notuvia-mcp`:

* `create_note(type, title, body, taxonomies?)` — mint a ULID, write to the hex-sharded path (`notes/<aa>/<bb>/<id>.md`, ADR 0004) with valid frontmatter (`id`, `type`, `created`, `title`), validate taxonomy keys/values against the registry.
* `update_note(id, ...)` — modify title, body, and taxonomy values. Reserved core fields (`id`, `created`, `encrypted`, sharding) are not writable. **Reject body updates to encrypted notes** — the server can't re-seal without the key (ADR 0028); metadata updates on encrypted notes preserve the envelope untouched, matching the app's locked-note guard.
* `delete_note(id)` — remove the note file (and reconcile the index). Attachments under `attachments/<id>/` handled the same way the app's delete does.

After each write, reconcile the MCP process's own index connection for the touched path (same `reconcile_path()` used by the watcher). The running app picks the change up via its watcher within ~400ms — the same proven path as git-sync pulls; self-write suppression is keyed on content hash in the app's process, so external writes are not suppressed.

Concurrency: WAL + the existing 5s busy timeout covers two writers; last-write-wins on the file itself is acceptable for a single-user vault (same stance as git-sync, ADR 0013).
