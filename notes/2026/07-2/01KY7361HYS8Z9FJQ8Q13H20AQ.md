Scope raised from the original field-reordering bug (kept below as a finding): the sync pipeline has now produced four incident classes in three days — the ISE-73 comment stranding (DEV-991/995/997), the ISE-118 renumbering (DEV-996), push-race error flashes (DEV-1004), and today's reorder ping-pong and stamping race. Every fix so far has been point-wise. This issue is a **fresh-eyes investigation of the end-to-end process** — save → debounce → commit → push/pull → merge → conflict resolution → index reconcile, across both writers (app and MCP sidecar) — asking why the merge is brittle *structurally* and what end-state design makes conflicts rare and their resolution boring.

**Deliverable:** an ADR proposing the target design, plus a staged set of implementation issues. Not code.

**Avenues to examine (suggestions, not conclusions):**

* **Line-level git merge of YAML is the root fragility.** Two writers serialising the same semantic note differently (field order, precision, quoting) conflict without disagreeing. Should note merges be *semantic* — a three-way field-level merge in `resolve_conflicts` (base/ours/theirs parsed, per-field resolution, comments as an append-only union) — with git's line merge only a fast path?
* **Canonical serialisation as a contract**: if every writer (app, sidecar, import, future clients) emitted byte-identical serialisation for identical content — fixed field order, fixed timestamp precision — a whole class of pseudo-conflicts disappears. What would enforcing that cost?
* **Does** `updated` **belong in the file?** It has driven three incidents. Could it be derived (file mtime / git) or excluded from merge comparison everywhere by design rather than by special cases?
* **Write funnelling**: the app and sidecar both write whole notes. Are there flows (comments, status changes, sprint assignment) that should be modelled as narrower operations so concurrent edits compose instead of colliding?
* **Keep-both as last resort**: today it fires on anything the point-fixes don't catch. What's the complete taxonomy of divergence cases, and which genuinely need a human?
* **Verification**: a concurrent-writers chaos harness (two runtimes, scripted interleavings) so merge behaviour is tested as a system, not per-incident.
* **Observability**: conflict/retry counts surfaced somewhere reviewable, so brittleness shows up as a trend rather than as user-visible incidents.

**Original finding (2026-07-19, kept as investigation input):** `update_note`**'s field reordering escapes the no-op save guard.**

Observed ~14:23 during conflict forensics: the Mac app rewrote two ISE-agent tasks with diffs that were purely an `updated:` bump plus the `sprint:` line moving position — values identical. Those content-free rewrites raced the agent's real edits and contributed to keep-both conflict copies.

Mechanism: DEV-995's guard compares the note serialised at load vs after mutation. But `update_note` clears and re-sets managed fields (`milestones`/`sprints`/`milestone`/`sprint`, and the taxonomy loop), which moves them to the end of the frontmatter mapping. For a note whose on-disk field order differs from `update_note`'s canonical output — e.g. any note created by the MCP sidecar — the two serialisations differ even when every value is identical, so the guard sees a change and writes. Net effect: merely opening an MCP-authored note in the app rewrites it, and the app and sidecar can ping-pong reorder writes at each other. DEV-997's one-sided resolution doesn't catch it, because a reorder isn't timestamp-only. The point-fix would be an order-insensitive (semantic) no-op comparison — but see the canonical-serialisation and semantic-merge avenues above for the structural version of the same idea.

Related: DEV-991, DEV-995, DEV-996, DEV-997, DEV-1004 — the point-fix trail this investigation should subsume.
