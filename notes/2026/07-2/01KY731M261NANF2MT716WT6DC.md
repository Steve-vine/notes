Design follow-up from the 2026-07-17 ISE conflict incident (DEV-991). Three structural gaps in the DEV-615 keep-both model remain after that fix:

1. **Any concurrent same-note write conflicts, even when the edits are disjoint.** Every save rewrites the `updated:` frontmatter line, so two machines touching the same note always conflict in git regardless of what actually changed. A field-aware merge (frontmatter as a keyed map: take the later `updated`, merge edits to different keys, only keep-both on a genuine same-field content collision — the DEV-984/DEV-991 counter handling generalised) would have resolved the incident with no copies at all.
2. **Both sides mint a conflict copy.** Resolution is symmetric — each machine keeps its own version and writes the remote as a copy under a fresh ULID — so every real conflict yields two copy notes that both survive the cross-merge. Either only one deterministic side should mint (e.g. the lexically-smaller machine id), or the copy's id should be derived deterministically from (original id, conflicting content hash) so both machines mint the same note and it dedupes on merge.
3. **Restoring a trashed project whose identifier has since been claimed.** Once DEV-990 makes trashed projects release their identifier, a restore can silently reintroduce a duplicate. The restore path should detect the collision and clear or prompt for the identifier.
