    **One real bug worth recording.** Coverage has to be measured over the entities an integration *knows* — from its `entity_alias` rows — not over the entities it has *tagged*. My first implementation derived the denominator from tags, which silently drops the untagged host: DataDog reported 100% coverage while half its hosts carried no `env` at all. That's the exact failure the report exists to catch, so it would have been a bad one to ship. Two tests pin it now.

    **Three problems kept apart rather than merged into a hygiene score**, because they need different actions from different people: missing (someone must go and label things), non-standard-but-working (free to leave, cheap to tidy), and unlisted-under-a-defined-key (the real non-compliance — resolves to nothing, no rule matches it). A single percentage would hide the distinction that decides what to do.

    **Presentation choices**: worst first (an unlisted value on forty entities outranks a spelling variant on one); raw counts beside the bar so nobody has to take a percentage on trust; each value links to its drilldown; long lists cap at ten and say how many more. - **Report endpoint + Tag Cloud hygiene view** (all users, not admin-only) showing, per integration:
  - coverage % per governed key over its expected entity types ("DataDog hosts: 60% carry `env`");
  - values resolving only via alias (working, but non-standard at source);
  - values under `defined` keys that exist in no dictionary list (true non-compliance, with entity counts and drill-through to the entities);
  - pending mapping candidates (link into the Proposals queue).
