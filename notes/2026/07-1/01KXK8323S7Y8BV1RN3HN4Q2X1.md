The vendor detail page and the flag manage/assign UI, modelled on `RiskDetailPage.tsx` (ADR 0039 §3, §9).

- [x] `VendorDetailPage.tsx`: header with state + compliance-status (+ criticality) pills; editable details card (name, website, description, notes, criticality, owner) with the `updated_at`-keyed reset pattern, inputs disabled without canWriteVendors; replaces the `/vendors/:id` Placeholder.
- [x] State-transition control (Lifecycle card): buttons generated from `src/vendors/transitions.ts` — a client mirror of the model's map — offering only the allowed next states (server remains the enforcement boundary); terminal-offboarded notice with admin-only "Reinstate as active" override.
- [x] Flags: assign/unassign via MultiSelect (PATCHes `flag_ids` on change; read-only roles see badges) + a manage-flags modal (create with colour picker, inline rename/recolour, hard delete) gated on canWriteVendors. Flag mutations invalidate vendor list/detail queries (flags are embedded on vendor payloads).
- [x] Revision history tab (from `GET /vendors/{id}/revisions`, newest first) + `ActivityHistory` (admin-only internally, entityType vendors).
- [x] New hooks: `useVendorRevisions`, `useCreateVendorFlag`, `useUpdateVendorFlag`, `useDeleteVendorFlag`.
- [x] Vitest: detail renders, transition control shows only legal states, state + details PATCH bodies captured, read-only gating, admin reinstate, history tab (7 tests; suite 171 green).
