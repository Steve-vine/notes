The incident assignee picker currently offers **every enabled user** and labels them by **email**; the incident list's Assignee column also shows the email. Change both:

1. **Only Operators appear in the picker.** `/api/v1/users/assignable` (`users.py:53`) filters on `disabled` only — add a role filter using the ROLE_ORDER hierarchy (`auth/rbac.py:15`, viewer < operator < approver < admin): assignable = enabled users whose role level ≥ operator, so approvers/admins remain assignable (they *are* operators+), viewers drop out. Roles live in the `User.roles` JSONB list — filter in Python after the query or via JSONB containment; unknown roles confer nothing (`role_level`).
2. **Picker shows full name.** `AssignableUser` already carries `name` (`schemas.py:204-210`) — the frontend Select just labels with `u.email` (`IssueDetailPage.tsx:306`). Label with `u.name`; keep `searchable` matching on name (optionally show email as the option description to disambiguate duplicate names). Also the viewer's read-only line (`Assignee: {assignee_email}`, ~:293).
3. **Incident list shows full name.** `IssuesPage.tsx:191-192` renders `issue.assignee_email`. Add a denormalised `assignee_name` to `IssueRead` (same pattern as the existing `assignee_email` property, `models.py:458-462`) and render that; keep `assignee_email` on the schema for compatibility. Regenerate the OpenAPI types.

Note: users mirror EntraID (ADR 0015) — roles are read from the mirror, never edited in ISE. A user already assigned who later loses operator (or is a viewer today) keeps the assignment; the filter governs who *new* work can be handed to, matching the existing disabled-user behaviour.

Related: ISE-160/168 established full-name display on the timeline — this completes the email→name cleanup on the incident surfaces.
