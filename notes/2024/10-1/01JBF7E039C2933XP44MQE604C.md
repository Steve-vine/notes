### Create a new SSO Profile

Create an initial profile tied to your admin login for AWS
```
aws configure sso --profile <profile-name>
```
Make the profile name short and easy to remember and type in e.g. 's.vine'

You'll go through the following settings:

```
SSO session name (Recommended): <session-name>
```
Session name should be unique but able to tie to the profile e.g. 's.vine-session'

```
SSO start URL [None]: https://d-9c67641241.awsapps.com/start
```
This is the URL of AWS Access Portal

```
SSO region [None]: <region>
```
Default region to start in e.g. 'eu-west-2' for London

```
SSO registration scopes [sso:account:access]:
```
Accept the default (hit return)

Next it will open a browser session to authenticate you, follow the ion-screen instructions
```
Attempting to automatically open the SSO authorization page in your default browser.
If the browser does not open or you wish to use a different device to authorize this request, open the following URL:

https://device.sso.eu-west-2.amazonaws.com/

Then enter the code:
```

Once authenticated you will need to select the AWS Account from the list available
Select the 'Org Admin' account

You should now be logged in.

### Check and Re-validate a Session 

You can check the SSO Profile is valid by typing
```
aws sts get-caller-identity --profile <profile-name>
```
This should provide the currently logged in user details
```
{
    "UserId": "AROAQIJRRZFYT3D7QR36I:S.Vine@moneypenny.co.uk",
    "Account": "017820666225",
    "Arn": "arn:aws:sts::017820666225:assumed-role/AWSReservedSSO_AdministratorAccess_6241051e8e63225e/S.Vine@moneypenny.co.uk"
}
```

If the current login has expired you'll see this message
```
Error when retrieving token from sso: Token has expired and refresh failed
```

The session can be revalidated using the following command
```
aws sso login --sso-session <session-name>
```
This will prompt for an SSO browser logon again
Note:  This is re-validating the current session not creating a new profile so the session name needs to be provided e.g. 's.vine-session'

### Using Profiles

To list all available profiles
```
aws configure list-profiles
```

To specify the profile used when executing a command, suffix the --profile switch to the end of a command
```
aws s3 ls --profile <profile-name>
```

To set the default profile, set the environment variable
```
export AWS_PROFILE="<profile-name>"
```
Unset the variable to clear the default profile

If you try to run a command without specifying the --profile switch and no default profile is set you will get
```
Unable to locate credentials. You can configure credentials by running "aws configure".
```

To create a profile using long-term credentials such as access keys (Access Key ID and Secret Access Key)
```
aws configure --profile <profile-name>
```
**Note:  Use of long term credentials is no-longer recommended by AWS due to security concerns and should only be employed in situations where the use of temporary access token cannot be employed.**

Manual configuration of profiles can be completed by editing the configuration file:
~/.aws/config 

### Assume Another Role
In order to perform actions that require specific permissions, it may be required to assume a different role.

When assuming a role you still need to be authenticated through an active profile token so will either need to specify the --profile switch or set the AWS_PROFILE environmental variable.
#### Option 1 - Manually Assume a Role

Authorise the session and obtain the session details
```
aws sts assume-role --role-arn arn:aws:iam::<account-number>:role/<role> --role-session-name <session-name>
```
You will specify the AWS account number and role within that account and assign a session name to be refer to the session.

This will return the details to be use within the temporary session as below
```
{
    "Credentials": {
        "AccessKeyId": "ASIAXNGUVJ5ULGCMVBYH",
        "SecretAccessKey": “xxx",
        "SessionToken": “xxx",
        "Expiration": "2024-10-31T15:06:42+00:00"
    },
    "AssumedRoleUser": {
        "AssumedRoleId": "AROAXNGUVJ5UD4ODCC76U:MySession",
        "Arn": "arn:aws:sts::509399617384:assumed-role/EKSClusterAdminRole/MySession"
    }
}
```

Assign the AccessKeyId, AescretAccessKey and SessionToken to environmental variables
```
export AWS_ACCESS_KEY_ID="<AccessKeyId>"
export AWS_SECRET_ACCESS_KEY="<SecretAccessKey>"
export AWS_SESSION_TOKEN="<SessionToken>"
```

#### Option 2 - Assign With One Command

If using Linux/Mac you can perform the operation with a single command
```
eval $(aws sts assume-role --role-arn arn:aws:iam::<account-number>:role/<role> \
--role-session-name <session-name> \
--profile=<profile-name> \
--query 'Credentials.[AccessKeyId,SecretAccessKey,SessionToken]' \
--output text | awk '{print "export AWS_ACCESS_KEY_ID="$1"\nexport AWS_SECRET_ACCESS_KEY="$2"\nexport AWS_SESSION_TOKEN="$3}')
```

Example
```
eval $(aws sts assume-role --role-arn arn:aws:iam::183295409300:role/EKSClusterAdminRole \
--role-session-name eksClusterAdminRole-session \
--profile=s.vine \
--query 'Credentials.[AccessKeyId,SecretAccessKey,SessionToken]' \
--output text | awk '{print "export AWS_ACCESS_KEY_ID="$1"\nexport AWS_SECRET_ACCESS_KEY="$2"\nexport AWS_SESSION_TOKEN="$3}')
```

Once you have assumed the role, this should be visible when you run 
```
aws sts get-caller-identity
```

To stop using the assumed role, just clear the environmental variables
```
unset AWS_ACCESS_KEY_ID
unset AWS_SECRET_ACCESS_KEY
unset AWS_SESSION_TOKEN
```
